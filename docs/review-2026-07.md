## 1. Architecture summary

```
APSC S3 bucket (public PDFs, 2020→)
      │  01_download.py  (HEAD-probe business days, PS-number ±2 offsets)
      ▼
data/pdfs/*.pdf ──── r2_sync.py push-pdfs ───► private R2 (PDF archive)
      │  02_parse.py batch  (pdftotext → regex field extraction; parse_log.csv
      │                      is the incremental ledger; raw_vn_count guard)
      ▼
gazette_vacancies_raw.parquet          (170,512 rows: dailies + weeklies, pre-dedup)
      │  03_normalise.py   (salary_min/max + guards 1–6, closing_date,
      │                     location_normalised, classification_clean, duties_text)
      ▼
gazette_vacancies_normalised.parquet
      │  05_apply_crosswalk.py  (agency_canonical/agency_group; 3 passes:
      │      normal / .pdf-agency / division-required; uses committed
      │      data/agency_crosswalk.csv + live match_canonical() from
      │      04_build_crosswalk.py, which is regenerated manually)
      ▼
gazette_vacancies_crosswalk.parquet
      │  06_build_release.py  (drop test notices, daily-vs-weekly dedup,
      │                        rename cols, runtime validation assertions)
      ▼
data/release/aps_gazette_vacancies.parquet
      │  07_clean_text.py  (frequency-based per-agency boilerplate strip →
      │                     description_clean; PII redaction; rewrites release IN PLACE)
      │  08_classify_job_family.py  (claude-sonnet-4-6, prompt 2025-v2,
      │                     Batch/sync API; joins from private
      │                     job_family_classifications.parquet; rewrites release IN PLACE)
      │  09_check_coverage.py  (warn-only crosswalk-gap report)
      ▼
r2_sync.py push ──► public R2 (parquet + csv.gz) + private R2 (state)
```

CI: GitHub Actions weekly (Fri 01:00 UTC), R2 as the persistent state store. Incrementality is via `manifest.csv` (download), `parse_log.csv` (parse), and key-presence in `job_family_classifications.parquet` (classification).

**What's solid [V]:**
- Zero duplicate `vacancy_no` in the release; daily/weekly dedup works; no full-row duplicates.
- Salary guards all hold in the output (0 rows >$500k, 0 inverted ranges).
- Parse completeness: `record_count == vn_count` for **all 1,930 parsed PDFs** — no silently dropped notices.
- Agency crosswalk coverage is excellent: only **2** null `agency_canonical` rows (0.002%).
- Provenance for the classifier is better than most LLM pipelines: model, prompt version, raw response, and timestamp are stored per row in the private classifications parquet; a hand-labelled 200-ad validation set with committed comparison reports backs the accuracy claims.
- Documentation (README + data dictionary) is unusually honest and detailed; most known quirks are already written down.
- The 259-row ASIC/HoR "agency-website bleed" bug described in `TODO.md` appears **already fixed** (Option B + Mode C regexes in `02_parse.py:361-374`): the release now has only 25 null descriptions and 1 `agency_website` containing "Job Description". TODO.md is stale.

**What's fragile:**
- The whole parse is regex-over-`pdftotext` with no versioned regression fixtures and **no automated tests at all** — parser changes are validated only by runtime assertions in `06_build_release.py` and eyeballing.
- Portfolio-header resolution (`DIVISION_REQUIRED` / `PORTFOLIO_ONLY` / `BRANCH_OVERRIDE_PAIRS`) is a hand-maintained rule tower; each new gazette layout quirk needs a new rule, and failures are silent (row lands on the *portfolio host* agency, which is a plausible-looking wrong answer — exactly the AIFS failure).
- `description_clean` is recomputed globally on every run from frequency statistics, so published text can change retroactively with no version marker.
- Incremental state (parse log, manifest sentinels, classification keys) has a few "never retry" traps described below.

**Not determinable from code alone:** whether the published R2 objects match the local release (I could not access R2); the actual contents of the private classifications history before the June-2026 reclassify; pdftotext/poppler version used by past CI runs.

## 4. Implementation sketches (top 5 only — no code changed)

### R1. Publish AIFS fix + portfolio-mismatch guard
**Approach.** (1) Re-run `05 → 06 → 07 → 08` locally (or trigger `workflow_dispatch`) and `r2_sync.py push` so the published parquet picks up commit `7964b52`. (2) Add a validation to `06_build_release.py` (or `09_check_coverage.py`): for each row, if `division` starts with a canonical agency name (from the crosswalk CSV) that differs from `agency_canonical`, and the pair is not in an explicit allowlist (Federal Court↔FCFCoA/NNTT, the `BRANCH_OVERRIDE_PAIRS` hosts, portfolio departments), emit a warning table like the existing crosswalk-gap report.
**Files.** `06_build_release.py` or `09_check_coverage.py`; allowlist constant could live in `04_build_crosswalk.py` next to `BRANCH_OVERRIDE_PAIRS`.
**Backfill.** None — crosswalk application is a full recompute each run; the 116 rows fix themselves on rebuild. Note in the data dictionary changelog that pre-2026-07 downloads misattribute AIFS.
**Validation.** Assert AIFS ≥ 118 rows in `06` (same pattern as the eSafety check); confirm Services Australia count drops by ~116; run my division-mismatch scan (426 → all-allowlisted).
**Effort.** 2–4 hours.

### R2. Affirmative-measures linkage key
**Approach.** Add two opt-in columns in `06_build_release.py`, computed post-dedup: `is_affirmative_measure` (regex on `job_title`, `(?i)affirmative measure`) and `posting_group_id` = stable hash of (`agency_canonical`, AM-stripped+normalised title, `closing_date`) assigned only where a group contains >1 row or ≥1 AM row (else null, to avoid implying linkage that wasn't checked). Keep all rows; document "for role-level counts, count distinct `posting_group_id` (falling back to `vacancy_no` where null)".
**Files.** `06_build_release.py` (new derivation section), `docs/data_dictionary.md`.
**Backfill.** Automatic — computed from existing columns on every rebuild; no state file.
**Edge cases to test.** Title normalisation must not over-merge genuinely distinct roles with identical titles and closing dates within the same agency (my scan: 3,473 multi-groups total, only 1,586 AM-related — the other ~1,900 include bulk campaigns that may be genuinely distinct positions; hence flag AM groups confidently, non-AM groups conservatively or not at all). Watch AM phrasing variants: "(Affirmative Measures - Disability)", "Affirmative Measures Indigenous", trailing "- " separators.
**Validation.** AUSTRAC May-2025 collapses 34 → ~12 distinct groups; year-level dedup counts match my measured excess (2,248 rows); spot-check 20 random groups by eye.
**Effort.** 1–2 days including documentation.

### R3. Boilerplate versioning + global-template pass
**Approach.** (1) In `07_clean_text.py`, add a *global* bin pass: compute sentence frequencies across **all agencies** per half-year (same MIN_BIN logic at the corpus level); sentences exceeding, say, 20% of *all* ads in a bin (the "employed under other Acts" family will be at ~75%) are stripped for every agency, including sub-MIN_BIN ones. Keep the per-agency pass for agency-specific boilerplate. (2) Version it: add `BOILERPLATE_METHOD_VERSION = "2026-07-v2"` and write it into the parquet's key-value metadata (and the audit CSV); archive the audit CSV per run instead of overwriting (e.g. suffix with version or date).
**Files.** `07_clean_text.py`; `docs/data_dictionary.md` (`description_clean` method section).
**Backfill.** Automatic — 07 recomputes over the whole release each run. Note this will retroactively change `description_clean` for ~378+ rows; that is the point, and the version stamp makes it detectable.
**Validation.** Re-run my check: `clean_hit_%` for `under other Acts` should fall from 3.6–5.1% to ≈0 in 2025Q3+ while pre-2025 rows are unchanged; per-ad retention quantiles shouldn't move materially outside 2025+ (guards against over-stripping); diff a sample of small-agency ads before/after.
**Effort.** ~1 day. (A quarterly rather than half-year bin for the global pass would also sharpen the May-2025 break, at slight sparsity cost.)

### R4. Parse-log integrity
**Approach.** Two changes in `02_parse.py`: (1) move the parse-log append *after* the parquet write — buffer log rows per run and flush them together with the parquet (or write the parquet incrementally every N PDFs and flush that batch's log rows then), so the log never runs ahead of persisted data; (2) in `_load_parse_log()`, only treat `status == 'parsed'` as done, so `missing_pdf`/`error` rows retry on the next run (dedupe the log on rewrite or take last-status-per-filename).
**Files.** `02_parse.py` (`parse_gazette_batch`, `_load_parse_log`, `_append_parse_log`).
**Backfill.** One-off: delete the two `missing_pdf` rows for the PS25 June-2026 dailies from `parse_log.csv` state in R2 (or rely on the new retry logic), re-download those PDFs, re-run parse → normalise → …; verify whether their VNs were already captured via the 2026-06-25 weekly (if so, release row count won't change — record either way).
**Validation.** Kill a batch run mid-way in a scratch copy of `data/`; restart; confirm final parquet row count equals a clean single run. Confirm the two June dailies appear in `parse_log` as `parsed`.
**Effort.** Half a day. Note the crash-loss window has existed since inception; a one-off reconciliation (parse_log record_count sums vs raw parquet group-by-file counts) would confirm no historical loss — my totals matched (170,512 = 170,512), so history is clean **[V]**.

### R5. Machine-readable MoG lineage
**Approach.** Ship `data/agency_lineage.csv` (predecessor_canonical, successor_canonical, effective_date, relation ∈ {rename, merge, split}, optional weight) generated from the existing `MOG_CHANGES` dict (which is currently dead documentation) extended with split weights — e.g. DAWE→DAFF/DCCEEW and DESE→Education/DEWR weighted by successor headcount shares from APSED June-2023, clearly labelled as approximations. Publish it next to the release files. Separately, resolve the Antarctic inconsistency: either date-split the `Australian Antarctic Division` override (pre-2022-07 → DAWE, post → DCCEEW), or document it as a deliberate functional-continuity exception; today it contradicts the stated "DCCEEW gets its own group" rule.
**Files.** `04_build_crosswalk.py` (emit lineage CSV from `MOG_CHANGES`; date-aware Antarctic override needs a hook in `05_apply_crosswalk.py` similar to the existing pre-MoG DAWE date fallback at line 194), `r2_sync.py` (publish), README/data dictionary.
**Backfill.** Antarctic change moves 73 pre-2022 rows DCCEEW → DAWE ("Agriculture department" group) on rebuild — call this out in a changelog since it changes two published time series.
**Validation.** DCCEEW pre-2022-07 count drops 75 → ~2 (the Environment-and-Energy overrides need the same decision); lineage CSV round-trips: every predecessor/successor is a valid canonical name; a demo notebook stitches the DCCEEW series using the weights.
**Effort.** 1–2 days, mostly deciding the weights and documenting their basis.

## 5. Questions for the maintainer

6. **`gazette_type='daily'` (456 rows):** these are daily-only notices with no weekly counterpart.
